## 4. Solidity File Structure & Contract Lifecycle



# 🟢 Part 1: Solidity File Structure

A Solidity file (`.sol`) is similar to a class file in OOP languages but designed for blockchain execution.

### ✅ Basic Structure of a Solidity File

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract MyContract {
    // State variables

    // Constructor

    // Functions

    // Events

    // Modifiers
}
```

---

## 🔹 1. SPDX License Identifier

```solidity
// SPDX-License-Identifier: MIT
```

 Specifies the license of the smart contract
 Required for publishing on platforms like Etherscan
 Helps avoid legal ambiguity

---

## 🔹 2. Pragma Directive

```solidity
pragma solidity ^0.8.0;
```

 Defines compiler version
 `^0.8.0` → compatible with versions `>=0.8.0 <0.9.0`
 Prevents unexpected behavior due to version mismatch

---

## 🔹 3. Contract Definition

```solidity
contract MyContract {
```

 Entry point of your smart contract
 Similar to a class in JavaScript or Java

---

## 🔹 4. State Variables

```solidity
uint public count;
address owner;
```

 Stored permanently on blockchain (in storage)
 Cost gas to modify
 Can have visibility: `public`, `private`, `internal`

---

## 🔹 5. Constructor

```solidity
constructor() {
    owner = msg.sender;
}
```

 Runs only once during deployment
 Used for initialization

---

## 🔹 6. Functions

```solidity
function increment() public {
    count++;
}
```

Types of functions:

 `view` → read-only
 `pure` → no state access
 `payable` → accepts ETH

---

## 🔹 7. Events

```solidity
event Increment(address user, uint value);
```

 Used for logging (important for frontend + debugging)
 Stored in transaction logs, not contract storage

---

## 🔹 8. Modifiers

```solidity
modifier onlyOwner() {
    require(msg.sender == owner, "Not owner");
    _;
}
```

 Used to reuse conditions (like access control)

---

## 🔹 9. Imports (Optional)

```solidity
import "./AnotherContract.sol";
```

 Helps split contracts into multiple files
 Important for large projects

---

# 🟢 Part 2: Contract Lifecycle (VERY IMPORTANT)

This is where most beginners struggle — understanding what happens from writing code → execution on blockchain.

---

## 🔁 Lifecycle Stages

### 1. Writing the Contract

 You write `.sol` code in tools like:

   Remix IDE
   Hardhat
   Foundry

---

### 2. Compilation

 Solidity code → converted into:

   Bytecode (runs on EVM)
   ABI (Application Binary Interface)

👉 ABI is crucial:

 Frontend uses it to interact with contract

---

### 3. Deployment

 Contract is deployed to blockchain via a transaction
 Requires gas (ETH)

👉 After deployment:

 Contract gets a unique address
 Stored permanently on blockchain

---

### 4. Execution (Interaction Phase)

Users interact with contract via:

#### 🔹 a. Read Functions (Free)

 `view`, `pure`
 No gas cost
 No blockchain state change

#### 🔹 b. Write Functions (Paid)

 Modify state
 Require gas
 Create transactions

---

### 5. State Changes

 Stored in Ethereum storage
 Persistent and immutable (unless coded otherwise)

---

### 6. Destruction (Optional)

```solidity
selfdestruct(payable(msg.sender));
```

 Deletes contract from blockchain (mostly deprecated now)
 Sends remaining ETH to address

---

# 🟢 Full Lifecycle Flow (Simplified)

```
Write → Compile → Deploy → Interact → Update State → (Optional) Destroy
```
