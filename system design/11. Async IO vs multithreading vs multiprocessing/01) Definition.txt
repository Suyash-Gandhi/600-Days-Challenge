

# 11. Async I/O vs Multithreading vs Multiprocessing

## 1. First, the Core Problem

When building systems (APIs, servers, etc.), you often need to handle multiple tasks at the same time:

 1000 users hitting your API
 Multiple DB calls
 External API requests
 File reads/writes

👉 The question is:
How do we handle concurrency efficiently?

There are 3 major approaches:

1. Async I/O (Event-driven)
2. Multithreading (Shared memory)
3. Multiprocessing (Separate memory)

---

# 2. Async I/O (Asynchronous Programming)

## 🔹 Idea

Instead of waiting (blocking), the system moves on to another task and comes back later.

👉 “Don’t wait → continue working → handle result when ready”

---

## 🔹 How It Works

 Uses event loop
 Uses non-blocking I/O
 Tasks are scheduled as coroutines

Example flow:

1. Send DB request
2. Don’t wait
3. Handle another request
4. When DB responds → resume

---

## 🔹 Real-Life Analogy

👉 You order food and don’t stand at the counter
You sit → do something else → pick food when ready

---

## 🔹 Best For

 I/O-bound tasks:

   API calls
   DB queries
   File operations
   Network requests

---

## 🔹 Pros

 Very lightweight
 High scalability (can handle thousands of connections)
 Efficient for I/O-heavy systems

---

## 🔹 Cons

 Not good for CPU-heavy tasks
 Harder to debug
 Requires async-compatible libraries

---

## 🔹 Example Use Cases

 Node.js servers
 FastAPI (Python async)
 Chat apps, streaming, real-time APIs

---

# 3. Multithreading

## 🔹 Idea

Run multiple threads inside a single process, sharing memory.

👉 “Multiple workers inside one room”

---

## 🔹 How It Works

 Threads share:

   Memory
   Variables
 OS schedules threads

---

## 🔹 Real-Life Analogy

👉 Multiple chefs in the same kitchen
They share ingredients (memory)

---

## 🔹 Best For

 Light parallel tasks
 Mixed workloads (I/O + small CPU)

---

## 🔹 Pros

 Faster than processes (less overhead)
 Shared memory → easy communication

---

## 🔹 Cons

 Race conditions
 Deadlocks
 Hard debugging
 Limited by GIL (in Python)

---

## 🔹 Example Use Cases

 Background tasks
 Web servers (thread pools)
 UI applications

---

# 4. Multiprocessing

## 🔹 Idea

Run multiple processes, each with separate memory space.

👉 “Multiple independent workers”

---

## 🔹 How It Works

 Each process:

   Has its own memory
   Runs independently
 Communication via:

   IPC (pipes, queues)

---

## 🔹 Real-Life Analogy

👉 Different kitchens working independently
(No sharing → no conflicts)

---

## 🔹 Best For

 CPU-bound tasks:

   Image processing
   ML computations
   Data processing

---

## 🔹 Pros

 True parallelism
 No GIL limitation
 More stable (fault isolation)

---

## 🔹 Cons

 High memory usage
 Slower communication
 Process creation overhead

---

## 🔹 Example Use Cases

 ML training
 Video encoding
 Heavy computations

---

# 5. Key Differences (VERY IMPORTANT)

| Feature     | Async I/O                    | Multithreading | Multiprocessing |
| ----------- | ---------------------------- | -------------- | --------------- |
| Model       | Event loop                   | Threads        | Processes       |
| Parallelism | ❌ (concurrent, not parallel) | ⚠️ Limited     | ✅ True parallel |
| Memory      | Shared (single thread)       | Shared         | Separate        |
| Best For    | I/O-bound                    | Mixed          | CPU-bound       |
| Overhead    | Very low                     | Medium         | High            |
| Complexity  | Medium                       | High           | High            |
| Scalability | Very high                    | Medium         | Medium          |

---

# 6. When to Use What (INTERVIEW GOLD)

## ✅ Use Async I/O when:

 You have many I/O operations
 Example:

   Chat server
   API gateway
   Web server

👉 This is why Node.js is async-first

---

## ✅ Use Multithreading when:

 Tasks need shared memory
 Light parallel work
 Example:

   Web server worker threads
   Background jobs

---

## ✅ Use Multiprocessing when:

 Heavy CPU work
 Need true parallelism
 Example:

   ML pipelines
   Data processing

---

# 7. Real System Design Mapping

## Example: Instagram / Chat App

 API handling → Async I/O
 Notifications → Multithreading
 Image processing → Multiprocessing

👉 Real systems use a combination of all three

---

# 8. Common Interview Trap

👉 Question: “Why not use multithreading everywhere?”

Answer:

 Threads cause race conditions
 Limited scalability
 Not efficient for massive I/O systems

👉 That’s why modern systems prefer:

 Async (for I/O)
 Multiprocessing (for CPU)
