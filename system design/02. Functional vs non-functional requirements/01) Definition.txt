## 2. Functional vs Non-Functional Requirements


# 🔹 1. Functional Requirements (FR)

### 👉 Definition

Functional requirements describe what the system should do — the actual features and behaviors.

They define:

 Inputs → Processing → Outputs
 Business logic
 User interactions

---

### 👉 Examples (Real Systems)

#### 📌 Instagram

 User can upload photos/videos
 User can like and comment
 User can follow/unfollow users

#### 📌 Amazon

 User can search products
 User can add items to cart
 User can place orders

---

### 👉 Key Characteristics

 Feature-specific
 Clearly defined actions
 Usually derived from business requirements
 Easy to test (works / doesn’t work)

---

### 👉 Example (Structured)

For a Chat Application:

 User can send messages
 User can receive messages in real-time
 User can create groups

---

# 🔹 2. Non-Functional Requirements (NFR)

### 👉 Definition

Non-functional requirements describe how well the system performs those functions.

They define:

 Performance
 Scalability
 Reliability
 Security
 Usability

---

### 👉 Examples (Real Systems)

#### 📌 Instagram

 Feed should load in < 2 seconds
 System should support millions of users
 High availability (minimal downtime)

#### 📌 Amazon

 Checkout must be secure
 Website should handle Black Friday traffic spikes
 Low latency for product search

---

### 👉 Key Characteristics

 Quality attributes
 System-wide impact
 Harder to measure precisely
 Crucial for large-scale systems

---

### 👉 Example (Structured)

For a Chat Application:

 Messages should be delivered in < 100 ms
 System should support 1 million concurrent users
 Messages should not be lost (reliability)

---

# 🔹 3. Core Differences (VERY IMPORTANT)

| Aspect  | Functional Requirements  | Non-Functional Requirements      |
| ------- | ------------------------ | -------------------------------- |
| Focus   | What system does         | How system performs              |
| Nature  | Features                 | Quality attributes               |
| Scope   | Specific functionalities | Entire system behavior           |
| Testing | Yes/No (works or not)    | Measured (latency, uptime, etc.) |
| Example | “Send message”           | “Message delivery < 100ms”       |

---

# 🔹 4. How Interviewers Expect You to Use This

In system design interviews:

### Step 1: Clarify Functional Requirements

You should ask:

 What features are needed?
 What are users allowed to do?

👉 Example:

> “Users should be able to upload posts, follow users, and view feeds.”

---

### Step 2: Define Non-Functional Requirements

You should explicitly state:

 Scale (users, requests/sec)
 Latency expectations
 Availability requirements

👉 Example:

> “We should support 10M users with feed load time under 200ms and 99.99% availability.”
