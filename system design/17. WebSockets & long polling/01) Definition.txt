## 17. WebSockets & Long Polling



# 🔹 1. The Problem: Real-Time Communication

In traditional HTTP:

 Client sends request → server responds → connection closes
 Server cannot push data on its own

👉 This is inefficient for:

 Chat apps (WhatsApp)
 Live notifications (Instagram likes)
 Stock price updates
 Multiplayer games

So we need techniques for real-time or near real-time communication.

---

# 🔹 2. Long Polling

## 📌 What it is

Long polling is a hack on top of HTTP to simulate real-time updates.

---

## 🔄 How it works (step-by-step)

1. Client sends request → “Any new data?”
2. Server:

    If data available → responds immediately
    If NOT → keeps request open (waits)
3. When new data arrives → server responds
4. Client immediately sends another request

👉 This cycle repeats continuously

---

## 📊 Flow

Client → Request → Server (waits) → Response → Client → Repeat

---

## ✅ Advantages

 Works over normal HTTP (no special protocol)
 Easy to implement
 Compatible with all browsers and proxies

---

## ❌ Disadvantages

 High latency compared to WebSockets
 Repeated HTTP overhead (headers every request)
 Not truly real-time
 Inefficient at scale

---

## 📌 Use Cases

 Simple notification systems
 Systems where real-time is not critical
 Legacy systems

---

# 🔹 3. WebSockets

## 📌 What it is

WebSocket is a full-duplex, persistent connection protocol.

👉 Once connected:

 Client ↔ Server can send data anytime
 No need to repeatedly reconnect

---

## 🔄 How it works

### Step 1: Handshake (HTTP → WebSocket upgrade)

Client sends:

```
Upgrade: websocket
```

Server responds:

```
101 Switching Protocols
```

👉 Connection upgrades from HTTP → WebSocket

---

### Step 2: Persistent connection

 Connection stays open
 Data flows in both directions anytime

---

## 📊 Flow

Client ⇄ Server (continuous, real-time)

---

## ✅ Advantages

 True real-time communication
 Low latency
 Efficient (no repeated headers)
 Bi-directional communication

---

## ❌ Disadvantages

 More complex to implement
 Requires connection management
 Harder to scale (stateful connections)
 Load balancers need special handling

---

## 📌 Use Cases

 Chat apps (WhatsApp, Slack)
 Live dashboards
 Gaming
 Trading systems
 Collaborative tools (Google Docs)

---

# 🔹 4. Key Differences (Very Important)

| Feature         | Long Polling   | WebSockets     |
| --------------- | -------------- | -------------- |
| Connection type | Repeated HTTP  | Persistent     |
| Latency         | Medium         | Low            |
| Server push     | Simulated      | Native         |
| Overhead        | High           | Low            |
| Complexity      | Low            | High           |
| Scalability     | Easier         | Harder         |
| Real-time       | Near real-time | True real-time |

---

# 🔹 5. When to Use What (Interview Insight)

## ✅ Use Long Polling when:

 Traffic is low
 Real-time is not critical
 Simplicity is preferred
 You need HTTP compatibility

---

## ✅ Use WebSockets when:

 Real-time updates are critical
 High-frequency communication
 Bidirectional communication needed
 Performance matters
