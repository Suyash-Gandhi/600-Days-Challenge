## 15. Immutable Updates in State (Arrays & Objects)

In React, state must be treated as immutable.
This means:

> ❌ Never modify the existing state directly
> ✅ Always create a new copy and then update it

This is extremely important because React relies on reference comparison to detect changes and decide whether to re-render.

---

# 1️⃣ Why Immutability Is Important in React

React does shallow comparison of state and props.

If you mutate state directly:

```js
state.user.name = "Suyash"
```

React sees:

 Old object reference === New object reference
 So React assumes nothing changed
 No re-render happens

But if you create a new object:

```js
setUser({ ...user, name: "Suyash" })
```

Now:

 Old reference ≠ New reference
 React detects change
 Component re-renders

### 🔎 Key Rule:

React re-renders when the reference changes, not when the internal value changes.

---

# 2️⃣ Immutable Updates with Objects

### ❌ Wrong Way (Mutating)

```js
const [user, setUser] = useState({
  name: "John",
  age: 22
})

const updateAge = () => {
  user.age = 23        // Direct mutation ❌
  setUser(user)
}
```

This may NOT re-render properly.

---

### ✅ Correct Way (Using Spread Operator)

```js
const updateAge = () => {
  setUser({
    ...user,
    age: 23
  })
}
```

What happens here?

 `...user` copies all existing properties
 `age: 23` overrides the age
 A new object is created

---

### 🔥 Functional Update (Best Practice)

When new state depends on previous state:

```js
setUser(prevUser => ({
  ...prevUser,
  age: prevUser.age + 1
}))
```

This prevents bugs caused by asynchronous updates.

---

# 3️⃣ Immutable Updates with Arrays

Arrays are the most common place where beginners make mistakes.

---

## Adding Items

### ❌ Wrong

```js
items.push("New Item")
setItems(items)
```

---

### ✅ Correct

```js
setItems([...items, "New Item"])
```

This creates a new array with previous elements + new item.

---

## Removing Items

### ❌ Wrong

```js
items.splice(index, 1)
setItems(items)
```

---

### ✅ Correct (Using filter)

```js
setItems(items.filter((_, i) => i !== index))
```

`filter()` returns a new array.

---

## Updating an Item Inside Array

### Example: Update item by id

```js
setItems(items.map(item =>
  item.id === id
    ? { ...item, name: "Updated Name" }
    : item
))
```

Here:

 `map()` creates a new array
 We copy only the item we want to change
 Other items remain untouched

---

# 4️⃣ Nested State Updates (Important for Interviews)

When state is nested:

```js
const [user, setUser] = useState({
  name: "John",
  address: {
    city: "Delhi",
    pincode: 110001
  }
})
```

### ❌ Wrong

```js
user.address.city = "Mumbai"
setUser(user)
```

---

### ✅ Correct (Deep Copy Step-by-Step)

```js
setUser(prevUser => ({
  ...prevUser,
  address: {
    ...prevUser.address,
    city: "Mumbai"
  }
}))
```

Why?

Because:

 You must copy `user`
 Then copy `address`
 Then update `city`

If you don’t copy each level → mutation happens.

---

# 5️⃣ Common Immutable Methods for Arrays

| Operation | Mutating ❌          | Immutable ✅             |
| --------- | ------------------- | ----------------------- |
| Add       | push()              | spread `[...arr, item]` |
| Remove    | splice()            | filter()                |
| Update    | direct assign       | map()                   |
| Sort      | sort() (mutates)    | `[...arr].sort()`       |
| Reverse   | reverse() (mutates) | `[...arr].reverse()`    |

---

# 6️⃣ Advanced: Using Immer (Optional Concept)

Sometimes deep updates become messy.

Example library: Immer

```js
import { produce } from "immer"

setUser(produce(draft => {
  draft.address.city = "Mumbai"
}))
```

Immer allows writing "mutating-looking" code but keeps immutability internally.

---

# 7️⃣ Real-World Example (Todo App)

```js
const [todos, setTodos] = useState([
  { id: 1, text: "Learn React", completed: false }
])

const toggleTodo = id => {
  setTodos(prevTodos =>
    prevTodos.map(todo =>
      todo.id === id
        ? { ...todo, completed: !todo.completed }
        : todo
    )
  )
}
```

This:

 Avoids mutation
 Keeps state predictable
 Ensures proper re-render

---

# 8️⃣ Why Immutability Makes React Powerful

Immutability allows:

 Predictable UI updates
 Easier debugging
 Time-travel debugging (Redux)
 Performance optimization
 Safe concurrency (React 18)

Without immutability → React breaks.

---

# 9️⃣ Summary

### Core Rules:

1. Never modify state directly
2. Always create new arrays/objects
3. Copy every nested level that changes
4. Use functional updates when based on previous state

---

# 🔥 Interview-Level Explanation (Short Version)

> React state must be updated immutably because React relies on reference comparison to detect changes. Direct mutation keeps the same reference, preventing re-render. Therefore, we use spread operator, map, filter, and functional updates to create new copies while updating state.
