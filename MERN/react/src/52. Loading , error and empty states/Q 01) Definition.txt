

# 🔥 52. Loading / Error / Empty States in React

Whenever you fetch or process data, your UI should handle 3 major states:

### 1. Loading State

### 2. Error State

### 3. Empty State

These are called UI states for async data handling.

---

# 🧠 Why this matters

If you don’t handle these:

 User sees blank screen
 User thinks app is broken
 Poor UX → recruiters instantly notice this

A professional app ALWAYS handles these.

---

# 🧩 The 3 States Explained

---

## 1️⃣ Loading State

### 👉 What it means:

Data is being fetched, but not available yet.

### 👉 Example:

User opens page → API call starts → data not ready → show loader

---

### 💡 UI Options:

 Spinner
 Skeleton loading (better UX)
 "Loading..." text

---

### ✅ Example (React)

```js
const [loading, setLoading] = useState(true);

useEffect(() => {
  fetchData().then(() => {
    setLoading(false);
  });
}, []);

if (loading) {
  return <p>Loading...</p>;
}
```

---

### 🧠 Best Practice

 Prefer skeleton UI over spinner
 Avoid long blank waits

---

## 2️⃣ Error State

### 👉 What it means:

Something failed (API error, network issue, etc.)

---

### 👉 Example:

 Server down
 Wrong API URL
 Internet issue

---

### ✅ Example

```js
const [error, setError] = useState(null);

useEffect(() => {
  fetchData()
    .catch(() => setError("Something went wrong"));
}, []);

if (error) {
  return <p>{error}</p>;
}
```

---

### 🧠 Best Practice

Instead of:

> ❌ "Error occurred"

Use:

> ✅ "Failed to load data. Please try again."

Even better:

 Add Retry button

---

## 3️⃣ Empty State

### 👉 What it means:

Data loaded successfully, but no data exists

---

### 👉 Example:

 No tasks in todo app
 No search results
 No messages

---

### ✅ Example

```js
if (data.length === 0) {
  return <p>No items found</p>;
}
```

---

### 🧠 Best Practice

Instead of boring text:

> ❌ "No data"

Use:

> ✅ "No tasks yet. Create your first task!"

→ encourages action

---

# 🔄 Putting It All Together

### 💥 Real Flow

```js
if (loading) return <Loading />;
if (error) return <Error />;
if (data.length === 0) return <Empty />;

return <DataList data={data} />;
```

---

# 🧱 Clean Component Structure (Best Practice)

Instead of messy inline logic, create reusable components:

```js
<Loading />
<Error message="..." />
<Empty message="..." />
```

---

# 🎯 Real Project Example

### 📝 Todo App

| State   | UI                       |
| ------- | ------------------------ |
| Loading | Spinner                  |
| Error   | "Failed to fetch todos"  |
| Empty   | "No todos yet. Add one!" |
| Success | Show list                |

# ⚡ Advanced Improvements (Recruiter-level)

---

## ✅ 1. Skeleton Loading (Better UX)

Instead of spinner:

```js
<Skeleton height={20} count={5} />
```

Feels faster because layout is visible.

---

## ✅ 2. Retry Button

```js
<button onClick={fetchData}>Retry</button>
```

---

## ✅ 3. Global Handling (Axios / Fetch Wrapper)

Instead of repeating logic everywhere:

 Create API utility
 Handle errors globally

---

## ✅ 4. Combine States (Cleaner)

```js
const [status, setStatus] = useState("loading"); 
// loading | success | error
```

---

## ✅ 5. Libraries That Handle This Automatically

 React Query
 SWR

They give you:

```js
const { data, isLoading, error } = useQuery(...)
```

---

# 🚨 Common Mistakes

---

### ❌ 1. Not handling empty state

→ UI looks broken

---

### ❌ 2. Showing spinner forever

→ forgot to set loading false

---

### ❌ 3. Generic error messages

→ bad UX

---

### ❌ 4. Mixing all logic in JSX

→ unreadable code
